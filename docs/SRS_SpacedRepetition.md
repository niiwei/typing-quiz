# 间隔重复复习系统 - 功能方案文档

## 1. 资深记忆学习产品经理 (PRD)

### 功能名称：间隔重复复习系统 (Spaced Repetition System)

#### 核心概念映射

| Anki 概念 | 本系统映射 | 说明 |
|-----------|----------|------|
| 牌组 | 分组 | 用户的测验分组即为复习单元 |
| 卡片 | 测验 | 每个测验是一道"记忆卡片" |
| 学习阶段 | 首次学习 | 新测验需完成首次学习才能进入复习 |
| 复习阶段 | 间隔重复 | 根据算法安排复习时间 |
| 暂停/搁置 | 测验状态 | 暂停=不再安排；搁置=延迟复习 |

#### 功能特点

**1. 学习阶段 (Learning Phase)**
- 新测验默认进入"待学习"状态
- 首次打开测验即标记为"学习中"
- 完成一次打字/填空练习后，学习阶段结束
- 可选择"简单/正常/困难"评级，影响首次间隔

**2. 复习阶段 (Review Phase) - SM-2 算法简化版**

```
interval(新) = 初始间隔 × 难度系数
interval(复习) = 前次间隔 × 间隔倍数(基于评级)
```

- 评级：简单(4.0) / 正常(3.0) / 困难(2.0) / 重来(1.0)
- 间隔倍数：简单=2.5倍，正常=2.0倍，困难=1.5倍，重来=1天重置
- 初始间隔：1天，难度系数根据首次评级调整

**3. 测验状态管理**

| 状态 | 说明 |
|------|------|
| **待学习** | 新加入，从未练习过 |
| **学习中** | 正在首次学习 |
| **今日复习** | 算法安排到今天复习 |
| **待复习** | 已安排未来日期 |
| **搁置** | 用户主动延迟（可设1/3/7天后恢复） |
| **暂停** | 长期暂停，不纳入复习队列 |

**4. 推送机制**
- 每日首次登录显示"今日复习"弹窗
- 侧边栏显示各分组的复习数量角标
- 分组页面顶部显示"今日有 X 个测验待复习"

## 2. 资深用户体验经理

### 用户旅程

**场景：小明准备英语四级词汇复习**

```
Day 1 (导入新测验)
├─ 打开应用 → 侧边栏看到"英语四级"分组有 [5] 个待学习
├─ 点击进入分组 → 列表显示"待学习"标签
├─ 打开"abandon"测验 → 首次学习完成
├─ 练习后弹窗询问"掌握程度？"→ 选择"正常"
└─ 系统提示"3天后复习"→ 测验状态变为"待复习(3天后)"

Day 4 (首次复习)
├─ 打开应用 → 首页弹窗"英语四级有 1 个测验待复习"
├─ 一键进入 → 打开"abandon"测验
├─ 练习后评级"困难"→ 间隔缩短
└─ 测验状态变为"待复习(2天后)"

Day 6 (第二次复习)
├─ 侧边栏角标提示 → 进入快速复习
├─ 评级"简单"→ 间隔拉长到7天
└─ 感觉太简单，主动点击"暂停此测验"

Week 2 (批量复习)
├─ 首页显示"今日复习 12 个"
├─ 使用"批量复习模式"连续练习
├─ 完成全部 → 获得"今日复习完成"成就
└─ 查看统计图表，了解记忆曲线
```

### 解决的三个痛点

| 痛点 | 解决方案 |
|------|----------|
| **学了就忘** | 间隔重复算法在遗忘临界点推送复习，强化长期记忆 |
| **不知道该复习什么** | 自动计算每个测验的最佳复习时间，用户只需跟着系统走 |
| **新测验淹没在旧内容中** | 区分"学习"和"复习"阶段，优先完成新内容学习 |

### 预计障碍

1. **算法黑盒感** - 用户不理解为什么今天安排这5个测验
   - *缓解*：显示"上次练习日期"和"下次复习日期"

2. **学习负担不均** - 某天突然要复习50个，某天只有2个
   - *缓解*：提供"每日上限"设置，超限的自动顺延

3. **误操作惩罚** - 点了"重来"导致测验频繁出现
   - *缓解*：增加"修改上次评级"功能，允许用户纠错

## 3. 资深开发经理

### 技术栈选型

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 算法计算 | Java 服务端 | SM-2 算法在后台计算，避免前端泄露逻辑 |
| 状态存储 | MySQL + 新表 `quiz_review_status` | 与现有架构一致，便于查询今日复习列表 |
| 推送触发 | Spring Scheduler (每日 00:01) | 计算所有用户的"今日复习"，生成推送记录 |
| 前端提醒 | 原有 JS 弹窗 + LocalStorage | 记录"今日已推送"，避免重复打扰 |
| 复习队列 | 内存缓存 (可选 Redis) | 高频查询今日列表时的性能优化 |

### 运作逻辑

**1. 数据模型扩展**

```sql
quiz_review_status 表:
- quiz_id (FK)
- user_id (FK)
- status: enum(LEARNING, REVIEW, SUSPENDED, PAUSED)
- interval_days: int          -- 当前间隔天数
- ease_factor: float           -- 难度系数 (默认 2.5)
- next_review_date: date       -- 下次复习日期
- last_review_date: date       -- 上次复习日期
- review_count: int            -- 累计复习次数
```

**2. 核心算法流程 (每日任务)**

```
Scheduler @ 00:01:
  对每个用户:
    查询 next_review_date = 今天的测验
    更新状态 status = DUE_TODAY
    生成分组汇总统计
```

**3. 用户练习后的处理**

```
用户完成测验 + 评级:
  ├─ 若首次学习 → 设置 initial_interval
  ├─ 若复习阶段 → 计算新间隔 = 旧间隔 × ease_factor × rating_multiplier
  ├─ 更新 next_review_date = today + interval
  └─ 调整 ease_factor (评级好则↑，差则↓)
```

**4. 前端提醒机制**

```
页面加载时:
  调用 GET /api/review/today-summary
  返回: {分组名: 待复习数量}

  若有待复习:
    显示弹窗 / 侧边栏角标
    LocalStorage 标记 "pushed_today = true"
```

**5. 关键 API 端点**

| 端点 | 说明 |
|------|------|
| `GET /api/review/today` | 获取今日复习列表 |
| `POST /api/review/{quizId}/rate` | 提交评级，更新算法参数 |
| `POST /api/review/{quizId}/suspend` | 暂停/恢复测验 |
| `GET /api/review/stats` | 学习统计图表数据 |

**性能考虑**
- 每日 00:01 的调度任务对大数据量用户可能较慢 → 分批处理或异步队列
- 今日复习列表查询加索引 `(user_id, next_review_date)`
