# 项目文档指南

本文档介绍项目中所有规范文档的用途、目标读者、缺失后果及位置设计原因。

---

## 一、项目入口文档（根目录）

### 1. README.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 所有人（首次访问者、用户、开发者） |
| **主要功能** | 项目简介、功能特性、快速开始、技术栈 |
| **缺失后果** | 用户无法了解项目是什么、怎么用，失去第一印象 |
| **位置原因** | GitHub/GitLab 默认展示首页，必须一眼看到 |

---

### 2. CONTRIBUTING.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 开发者、贡献者 |
| **主要功能** | 如何参与开发、代码提交规范、Pull Request 流程 |
| **缺失后果** | 外部开发者不知如何贡献，代码质量参差不齐 |
| **位置原因** | GitHub 自动识别并显示"How to contribute"入口 |

---

### 3. ARCHITECTURE.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 开发者、架构师、新加入的团队成员 |
| **主要功能** | 系统架构图、技术选型、模块划分、数据流向 |
| **缺失后果** | 新成员难以理解系统设计，改动容易破坏架构 |
| **位置原因** | 技术核心文档，与 README 并列展示项目全貌 |

---

### 4. CHANGELOG.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 用户、开发者、运维人员 |
| **主要功能** | 版本发布历史、每个版本的变更、破坏性更新 |
| **缺失后果** | 升级时不知道改了什么，无法评估影响 |
| **位置原因** | 版本管理必备，根目录便于快速查阅历史 |

---

### 5. CODE_STYLE.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 开发者 |
| **主要功能** | 编码规范、命名约定、注释要求、提交信息格式 |
| **缺失后果** | 代码风格混乱，审查困难，维护成本增加 |
| **位置原因** | 与 CONTRIBUTING.md 配套，开发前必读 |

---

### 6. DEPLOYMENT.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 运维工程师、DevOps、项目负责人 |
| **主要功能** | 部署流程、环境配置、发布步骤、回滚方案 |
| **缺失后果** | 部署时手忙脚乱，容易出错，无法快速恢复 |
| **位置原因** | 运维人员最关心，需快速找到操作指南 |

---

### 7. SECURITY.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 安全研究人员、用户、开发者 |
| **主要功能** | 安全政策、漏洞报告方式、安全最佳实践 |
| **缺失后果** | 发现漏洞不知如何报告，安全风险无法及时修复 |
| **位置原因** | GitHub 自动显示安全政策，标准化安全流程 |

---

### 8. .env.example

| 项目 | 说明 |
|------|------|
| **目标读者** | 开发者、部署人员 |
| **主要功能** | 环境变量模板，展示需要配置哪些参数 |
| **缺失后果** | 新环境搭建困难，不知道要配哪些变量 |
| **位置原因** | 点前缀表示"隐藏/模板"，复制为 .env 后使用 |

---

## 二、技术细节文档（docs/目录）

### 9. docs/API.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 前后端开发者、第三方集成者 |
| **主要功能** | 接口定义、请求/响应格式、错误码、调用示例 |
| **缺失后果** | 前后端对接困难，接口使用混乱，集成成本高 |
| **位置原因** | 技术细节文档，根目录放摘要，详情放子目录 |

---

### 10. docs/DB_SCHEMA.md

| 项目 | 说明 |
|------|------|
| **目标读者** | 后端开发者、DBA、数据分析师 |
| **主要功能** | 表结构、字段说明、索引设计、关系图 |
| **缺失后果** | 数据库设计不清晰，查询效率低，维护困难 |
| **位置原因** | 数据库技术细节，与 API 文档并列归类 |

## 三、文档分层设计原则

```
根目录（项目级）     docs/子目录（技术级）
    ↓                     ↓
所有人必看            深入时查阅
快速了解全貌          详细技术细节
第一眼看到            按需阅读
```

| 层级 | 用途 | 放置文档类型 |
|------|------|--------------|
| **根目录** | 项目入口，快速导航 | 简介、规范、流程、安全 |
| **docs/** | 技术深入，详细参考 | API、数据库、架构细节 |
| **.env** | 配置模板 | 环境变量示例 |

## 四、文档优先级建议

### 新建项目时优先级

1. **P0 - 必须立即创建**
   - README.md（项目启动基石）

2. **P1 - 项目成型后创建**
   - ARCHITECTURE.md（技术方向明确后）
   - CONTRIBUTING.md（有外部贡献需求时）
   - CHANGELOG.md（第一个版本发布前）

3. **P2 - 团队扩大后创建**
   - CODE_STYLE.md（多人协作时）
   - DEPLOYMENT.md（正式环境部署时）
   - docs/API.md（接口稳定后）
   - docs/DB_SCHEMA.md（数据库定稿后）

4. **P3 - 可选补充**
   - SECURITY.md（公开项目/有安全要求时）
   - .env.example（有复杂环境配置时）

## 五、分场景精简建议

### 开源项目（GitHub/GitLab 公开）

| 文档 | 必要性 | 原因 |
|------|--------|------|
| CONTRIBUTING.md | **必须** | 外部贡献者需要知道如何参与 |
| README.md | **必须** | 第一印象，用户/开发者都看这 |
| GitHub 自动识别 | **重要** | 平台会显示特殊入口，降低参与门槛 |

### 个人闭源项目（自己开发）

| 文档 | 必要性 | 替代方案 |
|------|--------|----------|
| CONTRIBUTING.md | **不需要** | 你自己就是唯一开发者 |
| README.md | **必须** | 记录项目是啥、怎么运行（给自己看） |
| CODE_STYLE.md | **可选** | 可以只写在 README 里一句话 |
| GitHub 自动识别 | **不重要** | 反正没有外部贡献者 |

**最小文档集：**
```
README.md          ← 必须（项目简介 + 怎么运行）
ARCHITECTURE.md    ← 推荐（以后自己会忘）
DEPLOYMENT.md      ← 可选（部署步骤）
CHANGELOG.md       ← 可选（版本多了再补）
```

### 公司内部项目

| 文档 | 必要性 | 说明 |
|------|--------|------|
| CONTRIBUTING.md | **需要，但内容不同** | 写内部协作流程（Code Review 规则、分支策略），不是开源指南 |
| README.md | **必须** | 新成员入职、跨团队协作需要 |
| ARCHITECTURE.md | **强烈推荐** | 人员流动，文档是知识传承 |
| GitHub 自动识别 | **不重要** | 公司内部用 GitLab/私有仓库，功能入口不如开源社区重要 |

**最小文档集：**
```
README.md          ← 必须
ARCHITECTURE.md    ← 必须（团队知识共享）
CONTRIBUTING.md    ← 必须（内部协作规范，改名 INTERNAL_GUIDE.md 也行）
CODE_STYLE.md      ← 推荐（统一团队风格）
DEPLOYMENT.md      ← 必须（运维需要）
docs/API.md        ← 必须（前后端对接）
```

### 结论

"GitHub 自动识别"**只对开源项目重要**，闭源/内部项目可以大幅简化，**只保留真正有用的文档**。
